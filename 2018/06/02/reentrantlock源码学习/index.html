reentrantlock源码学习

发表于 2018-06-02 | 分类于 JDK | 阅读次数

没什么事，只是瞎写写

问题

  1. reentrantlock如何实现锁机制的

使用方式

reentrantLock.lock();
try{
    dosomething();
}finally {
    reentrantLock.unlock();
}

源码分析

lock

首先，我们先来看看lock方法都做了什么

public void lock() {
        sync.lock();
    }

可以看到实际上调用的是sync的lock方法，sync为ReentrantLock的一个内部类
一共有两种实现，一种是NonfairSync非公平锁，一种是FairSync公平锁，如果不指定的话，ReentrantLock默认是使用的非公平锁
我们先来看看NonfairSync的lock方法

if (compareAndSetState(0, 1))
    setExclusiveOwnerThread(Thread.currentThread());
else
    acquire(1);

因为NonfairSync是继承了aqs的,因此使用的是aqs中的compareAndSetState方法,尝试将state置为1，若成功则说明当前无其他线程
占用锁，当前线程自动获得锁，并将ExclusiveOwnerThread设置为当前线程
若抢占锁失败，则调用acquire方法，继续分享acquire方法

if (!tryAcquire(arg) &&
            acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
            selfInterrupt();

首先是调用tryAcquire方法尝试再次获得锁,因为我们使用的是NonfairSync，因此实际执行的方法为NonfairSync中的nonfairTryAcquire方法

final Thread current = Thread.currentThread();
            int c = getState();
            if (c == 0) {
                if (compareAndSetState(0, acquires)) {
                    setExclusiveOwnerThread(current);
                    return true;
                }
            }
            else if (current == getExclusiveOwnerThread()) {
                int nextc = c + acquires;
                if (nextc < 0) // overflow
                    throw new Error("Maximum lock count exceeded");
                setState(nextc);
                return true;
            }
            return false;

可以看到nonfairTryAcquire方法中依然是先尝试用cas操作去抢占锁，若失败的话，则判断当前尝试获取锁的线程是否为
ExclusiveOwnerThread，若是的话将当前state+1，这里的操作就是为称之为可重入式锁的原因

若tryAcquire方法执行失败，则调用acquireQueued方法将当前线程扔进队列里排队
我们先来看看addWaiter方法

Node node = new Node(Thread.currentThread(), mode);
       // Try the fast path of enq; backup to full enq on failure
       Node pred = tail;
       if (pred != null) {
           node.prev = pred;
           if (compareAndSetTail(pred, node)) {
               pred.next = node;
               return node;
           }
       }
       enq(node);
       return node;

主要完成的事是新建一个包含当前线程的节点，然后将此节点插入队尾插入成功之后返回这个节点
其中的enq方法的作用为循环执行上诉添加节点到队尾的逻辑，主要是防止并发场景下上一步添加节点操作失败
然后我们来查看acquireQueued方法

boolean failed = true;
        try {
            boolean interrupted = false;
            for (;;) {
                final Node p = node.predecessor();
                if (p == head && tryAcquire(arg)) {
                    setHead(node);
                    p.next = null; // help GC
                    failed = false;
                    return interrupted;
                }
                if (shouldParkAfterFailedAcquire(p, node) &&
                    parkAndCheckInterrupt())
                    interrupted = true;
            }
        } finally {
            if (failed)
                cancelAcquire(node);
        }

这里做的事情主要是循环检查当前节点是否为头节点之后的第一个节点，如果是的话则尝试抢占锁，若锁抢占成功
则将当前节点设置为头节点，并返回，否则的话便一直循环，其实至此一个非公平的可重入锁已经完全实现了,但是有个问题就是现在所有等待锁
的线程其实都在执行一个循环的操作，系统中运行的线程数并没有减少，这样会相当耗费资源，jdk是怎么解决这个问题的呢，接着往下分析

首先是shouldParkAfterFailedAcquire方法

int ws = pred.waitStatus;
if (ws == Node.SIGNAL)
    /*
     * This node has already set status asking a release
     * to signal it, so it can safely park.
     */
    return true;
if (ws > 0) {
    /*
     * Predecessor was cancelled. Skip over predecessors and
     * indicate retry.
     */
    do {
        node.prev = pred = pred.prev;
    } while (pred.waitStatus > 0);
    pred.next = node;
} else {
    /*
     * waitStatus must be 0 or PROPAGATE.  Indicate that we
     * need a signal, but don't park yet.  Caller will need to
     * retry to make sure it cannot acquire before parking.
     */
    compareAndSetWaitStatus(pred, ws, Node.SIGNAL);
}
return false;
  1. 若当前节点的前一个节点的状态为Node.SIGNAL,则说明前一个节点获得了一个信号量，即将被唤醒，因此当前节点的线程可以被阻塞掉，直接返回true
  2. 若前一个节点的状态为CANCEL，则从当前节点的前一个节点开始，依次去掉cancel掉的节点
  3. 若前一个节点的状态既不是SIGNAL也不是CANCEL,则将前一个节点的状态置为SIGNAL

若shouldParkAfterFailedAcquire方法返回了true，也就是场景1，那么接下来会执行
parkAndCheckInterrupt方法，此方法的目的就是将当前线程状态置为WAITING，此状态CPU不会为其分配时间片，需要显式地被其他线程唤醒

LockSupport.park(this);
return Thread.interrupted();

至此加锁的逻辑我们已经全部过完了，这是非公平锁的逻辑，公平锁的话大部分代码都相同
只是在cas操作前需要检查一下当前等待队列是否为空，接着来看看释放锁的逻辑

unlock

unlock方法调用的aqs的release方法

if (tryRelease(arg)) {
    Node h = head;
    if (h != null && h.waitStatus != 0)
        unparkSuccessor(h);
    return true;
}
return false;

tryRelease方法如下：

int c = getState() - releases;
            if (Thread.currentThread() != getExclusiveOwnerThread())
                throw new IllegalMonitorStateException();
            boolean free = false;
            if (c == 0) {
                free = true;
                setExclusiveOwnerThread(null);
            }
            setState(c);
            return free;

主要作用为释放当前线程占用的锁，若当前线程不占有锁了则unlock成功，将ExclusiveOwnerThread置为null
随后观察头结点当前的状态，若处于park状态，则调用unpark方法
